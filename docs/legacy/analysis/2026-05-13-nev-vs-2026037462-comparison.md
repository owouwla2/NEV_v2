## 3. 数据库与数据模型对比

### 3.1 NEV：MongoDB 单一主存

`@Document` 标注共 **16 个集合**：
- 用户：`users`（含 UserType: 消费者/商家/管理员/微信）、`auth_tokens`、`refresh_tokens`、`audit_logs`
- 电池溯源：`batteries`（traceNumber 唯一索引，含 `LifecycleRecord` 列表 + CarbonFootprint 引用）、`certifications`
- 商城：`merchants`、`products`、`carts`、`orders`、`payment_records`
- 碳：`carbon_footprints`、`carbon_credit_records`、`emission_factors`
- 其他：`knowledge_articles`、`trade_in_requests`
- 嵌入文档：`LifecycleRecord/CartItem/OrderItem/ShippingAddress/StatusLog/CarbonStage/...`

**特点**：
- 无 schema migration，索引由实体注解承担
- 文档模型嵌入度高（Battery 内嵌 LifecycleRecord 列表）
- 商城 demo seed 由 `DemoMallImportRunner` 一键导入（24 users / 8 merchants / 72 products / 40 orders，seed=20260325）

### 3.2 2026037462：MySQL + ClickHouse + Redis 多存储

**MySQL 业务表（25+，从 entity 和 mapper.xml 倒推）：**
- 电池主数据：`battery_label / battery_model / battery_structure / battery_structure_replacement / battery_production_info`
- 生命周期：`battery_lifecycle / quality_trace / cascade_inspection / recycle_order / recycle_inspection / remanufacturing_record`
- 召回与处置：`battery_recall / recall_execution / disposal_plan / disposal_result / anomaly_log`
- 决策引擎：`decision_engine_config / decision_execution / diversion_decision / maintenance_decision / material_price_snapshot / valuation_record`
- 区块链存证：`blockchain_evidence / BlockchainContractConfig`
- 系统扩展：`sys_dept_qualification / sys_field_pool / sys_template_definition / sys_template_field`

**ClickHouse（时序）**：`ChBatterySohTimeline`、`ChMaterialPriceHistory`
**Redis（缓存）**：Token 存储 + 通用缓存

**特点**：
- 关系型范式较严，召回-执行、决策-执行等都用主从表
- ClickHouse 专门承担电池 SOH 历史和原料价格时序，避免拖累主库
- 双数据源通过 `framework.aspectj.DataSourceAspect` AOP 切换

### 3.3 数据模型差异关键点

1. **电池实体粒度**：
   - NEV 的 `Battery` 是单文档，嵌入 `LifecycleRecord` 列表（适合一次性查整车）
   - 2026037462 拆为 `battery_production_info / battery_lifecycle / quality_trace / ...` 多表（适合按事件类型查询、批量分析）

2. **碳计算 vs 物料估值**：
   - NEV：`CarbonFootprint`（CF_raw + mfg + trans + use + eol）+ `EmissionFactor`
   - 2026037462：`MaterialPriceSnapshot` + `ValuationRecord`（残值评估）
   - 两个完全不同的维度，**互不冲突**，可共存

3. **角色体系**：
   - NEV：`UserType`（消费者/商家/管理员/微信）—— C 端导向
   - 2026037462：RuoYi `sys_role / sys_menu` + 业务 `sys_dept_qualification`（部门资质）—— B 端导向

## 4. 区块链与智能合约对比

| 维度 | NEV | 2026037462 |
|---|---|---|
| 链 | FISCO BCOS（4 节点 PBFT，WeBASE 部署） | FISCO BCOS（同） |
| 中继 | WeBASE-Front 5002 + node-mgr 5001 + web 5000 + sign 5004 | WeBASE-Front（路径脱敏） |
| Solidity 版本 | **0.4.25**（偏老） | **0.6.10** |
| 合约设计 | `Trace.sol` 主合约 + `BatteryInfoItem.sol` 每电池独立合约（gas 高） | `RoleManager` → `BatteryRegistry` → `LifecycleTrace`（3 层继承） |
| 角色合约 | `Producer/Distributor/Retailer + Roles.sol` | `RoleManager.sol` 单一来源（owner + roles 映射） |
| 数据上链 | `BatteryCreated` + `DistributorTraceAdded` + `RetailerTraceAdded` 3 事件 | 6 类事件：`PRODUCED / IN_USE / REPAIRED / RECYCLED / GRADIENT_UTILIZED / DISMANTLED`，含 `version+1` 不可改设计 |
| 链下链上一致性 | 由后端 service 保证 | 链下 SHA-256 dataHash → 上链 → 后续 `verifyBattery()` 双向校验 |
| 后端集成 | `nev.contract.TraceContract`（@PostConstruct 容错初始化，WeBASE 缺失时降级） | `client.ContractInvoker` + `BlockchainClient` + `BlockchainInitializer` + `BlockchainProperties` 四件套 |

**结论**：参考项目的合约**设计明显更先进**：dataHash 校验、版本号、6 类生命周期事件、AppendOnly 不可改。NEV 的"每电池一份独立合约"会随电池数线性增长部署成本。

## 6. 重构建议（按优先级排序）

### 🔴 P0 — 必做（数据模型 + 架构基础）

#### P0-1 数据模型统一策略：以 MongoDB 为主存，ClickHouse 补时序

**保留**：NEV 现有 16 个 MongoDB 集合作为业务核心存储
**引入**：ClickHouse 作为时序与分析侧库，处理：
- 电池 SOH 时间序列（按月/日聚合）
- 物料价格历史
- 大屏统计、报表

**理由**：
- MongoDB 文档模型已和 NEV 的 `Battery` 嵌入 `LifecycleRecord` 强耦合，强迁 MySQL 代价大
- ClickHouse 在时序聚合上比 MongoDB 快一个数量级，符合"大屏 + 分析"场景
- 拒绝引入 MySQL 作业务库，避免与 MongoDB 维度冲突

**落地**：
- 新增 `backend/src/main/java/nev/clickhouse/` 子包
- 参考 `2026037462/btld/btld-btld/src/main/java/com/zy/btld/config/ClickHouseConfig.java`
- 数据同步：在写 MongoDB 时通过 `ApplicationEventPublisher` 异步双写 ClickHouse

#### P0-2 后端分层加固（不上 Maven 多模块）

**不做**：拆 NEV 为多 Maven 模块（强行套 RuoYi 模板 ROI 不高）
**做**：在单模块内强化包分层 + 横切

- 把现有 `dao/`（15）+ `repository/`（1）统一改为 `repository/`，删除命名分裂
- 引入 AOP 注解（仿 RuoYi）：
  - `@nev.annotation.Log("操作描述")`：自动写 `audit_logs`
  - `@nev.annotation.RepeatSubmit`：防重复提交
  - `@nev.annotation.RateLimit`：限流
- 引入 `controller/admin/`、`controller/merchant/`、`controller/consumer/`、`controller/blockchain/` 子包，对应已有 admin-web 三套视图域

#### P0-3 智能合约升级到 Solidity 0.6.10 + AppendOnly 设计

**目标**：把 NEV 现有 `Trace.sol`（0.4.25）+ 每电池独立 `BatteryInfoItem.sol` 替换为参考项目的 3 层合约：
- `RoleManager.sol`：统一角色（替代当前 Producer/Distributor/Retailer 三合约）
- `BatteryRegistry.sol`：电池数字身份 + dataHash
- `LifecycleTrace.sol`：6 类事件（PRODUCED/IN_USE/REPAIRED/RECYCLED/GRADIENT_UTILIZED/DISMANTLED）+ 版本号

**理由**：
- 取消"每电池独立合约"，避免 gas 随电池数线性膨胀
- 引入 `dataHash` 校验，链下 MongoDB 与链上能双向核对
- 6 事件直接覆盖梯次利用 / 拆解，预留未来回收场景

**落地**：
- 在 `solidity/v3/contracts/` 新建（与 v2 并存，做 ABI 切换灰度）
- 后端 `nev.contract.TraceContract` 重命名为 `nev.contract.LifecycleTraceContract`，保留 v2 兼容方法
- 迁移完成后下线 v2

### 🟡 P1 — 推荐做（迁移参考项目的 B 端核心能力）

#### P1-1 移植"电池召回 + 召回执行"

- 新增集合：`battery_recalls`、`recall_executions`
- 新增 `nev.service.BatteryRecallService` + `nev.service.RecallExecutionService`
- admin-web 增加"召回管理"路由
- 链上：在 `LifecycleTrace.sol` 添加 `RECALLED` 事件类型

#### P1-2 移植"回收 + 梯次利用 + 再制造"闭环

- 新增集合：`recycle_orders`、`recycle_inspections`、`cascade_inspections`、`remanufacturing_records`
- 与 NEV 已有 `trade_in_requests`（以旧换新）对接：以旧换新接收后，进入 recycle_orders
- 链上事件：`RECYCLED` → `GRADIENT_UTILIZED`（梯次）→ `DISMANTLED`（拆解）→ `REMANUFACTURED`（新增）

#### P1-3 移植"决策引擎"

- 新增集合：`decision_engine_configs`（规则配置）、`decision_executions`（执行记录）
- 实现按规则触发：电池 SOH 低于阈值 → 触发"维修/分流/退役"决策
- admin-web 增加"决策中心"视图

#### P1-4 ClickHouse 大屏

- 用 NEV 现有 `Battery + CarbonFootprint + PaymentRecord` 数据作为同步源
- 新增 `apps/admin-web/src/views/dashboard/` 大屏视图
- 后端 `nev.controller.DashboardController` + `nev.clickhouse.AnalysisService`

### 🟢 P2 — 可选（提效与质量提升）

#### P2-1 引入代码生成器（轻量版）

- 不引入完整 `btld-generator` 模块（依赖 Velocity 模板太重）
- 写一个简化版 IDEA Live Template 或 npm 脚本：给定集合名 → 生成 controller + service + dao + entity 骨架

#### P2-2 移植"物料估值"

- 集合：`material_price_snapshots`、`valuation_records`
- 与 NEV 碳计算并存：碳排放看环境账，物料估值看经济账

#### P2-3 移植"合规监控"

- 接 `audit_logs` 已有数据，统计违规模式
- 与"决策引擎"形成联动（合规告警 → 自动决策）

#### P2-4 移植"模板字段池 / 动态表单"

- 适合"产品参数表"这种字段经常变的场景
- ROI 中等，可后期再做

### ⚪ P3 — 不建议做

- ❌ 整体迁移到 RuoYi 框架：业务模型不同，迁移成本大于收益
- ❌ 改用 MySQL 主库：MongoDB 已深度嵌入业务，无替换价值
- ❌ 把 2026037462 的代码生成器整套搬过来：依赖 Velocity，太重
- ❌ 移植 RuoYi 的菜单 + 字典系统：NEV 的前端路由 + 配置已经够用

## 8. 风险与注意事项

### 8.1 不要踩的坑

1. **不要混用两套用户体系**：NEV 的 `UserType`(消费者/商家/管理员/微信) 和 RuoYi 的 `sys_user/role/menu` 不要同时存在。保留 NEV 的，参考项目的菜单粒度仅用于设计参考。

3. **不要直接复制 RuoYi 的 Druid 配置**：NEV 用 MongoDB，多数据源需求弱，引入 Druid AOP 切换是过度工程。

4. **不要把 `application.properties` 的硬编码 ABI（line 31）作为升级蓝本**：v3 应严格走 env vars。
